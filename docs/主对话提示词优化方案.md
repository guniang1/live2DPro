# 主对话提示词优化方案（功能调整说明）

> 状态：**阶段 A/B + 身份区分已落地**（`PY/router/wschat.py`、`PY/utils/user_profile_refresh.py`，2026-05）  
> 关联代码：`PY/router/wschat.py`、`PY/live2d_db/memory_layers.py`、`PY/utils/live2d_catalog.py`  
> 编写依据：单条 System + 单条 User 架构评审结论（2026-05）

---

## 一、需求与目标

在**不推翻**现有「单条 system + 单条 user、历史折叠进记忆块」架构的前提下，降低上线后常见的：

1. 新旧记忆冲突（长期写「爱喝奶茶」，瞬时写「现在不喝」仍按长期答）
2. 人设漂移（长上下文后语气变「通用助手」）
3. 回复长短不稳定
4. 台词与 Live2D 表情/动作不匹配
5. 总上下文过长时的截断不可控

**非目标**：改回传统多轮 `user/assistant` messages 链；不改 Redis/MySQL 记忆写入链路（仅改注入 prompt 的拼装与预算）。

---

## 二、现状架构（基线）

### 2.1 主对话 Ollama messages

| 顺序 | role | 内容 |
|------|------|------|
| 1 | system | 五段拼接（见下） |
| 2 | user | `【当前用户问题】` + 本轮用户输入 |

历史**不**以独立 messages 轮次出现，而写入 system 内【三种记忆】。

### 2.2 system 五段顺序（当前）

```
【角色人设】     ← persona.tone_style + character_desc
【角色场景】     ← WebSocket scene_location
【用户人设】     ← user_profile（USER_PROFILE_IN_CHAT）
【用户时间】     ← WebSocket scene_time
【三种记忆】
  长期：…        ← Redis/MySQL period_overview
  短期：…        ← Redis 短期 List
  瞬时：…        ← Redis 瞬时 List（在 system 最末，紧贴下一条 user）
```

组装入口：`_build_memory_for_model` → `ollama_chat_messages`。

### 2.3 并行 Live2D 标签模型

- 与主对话**并行**请求（`OLLAMA_LIVE2D_TAGS_MODEL`，默认同 `OLLAMA_MODEL`）
- system：`catalog.action_llm_parallel_full_system(persona)`
- user：瞬时记忆 + 场景 + 【本轮用户输入】（**不含**本轮 assistant 回复）

### 2.4 已有长度控制（局部）

| 机制 | 环境变量 / 位置 | 说明 |
|------|-----------------|------|
| 长期注入截断 | `LONG_MEMORY_PROMPT_MAX_CHARS`（默认 2000） | `format_long_memory_block` **保留文尾** |
| 短期块上限 | `SHORT_TERM_PROMPT_MAX_CHARS`（默认 4000） | `format_short_term_block` |
| 瞬时轮数 | `INSTANT_MEMORY_MAX_TURNS`（默认 5） | Redis List |
| **缺失** | — | **组装后 system+user 总预算**、冲突优先级文案、角色锚点 |

### 2.5 设计意图（需保留）

访客路径注释（`wschat.py`）：画像/时间在前，【三种记忆】殿后，使**瞬时对话紧贴本轮 user**，减轻复述画像正文。

因此：**不建议**将整段 system 完全倒序（把人设挪到最后、瞬时挪到 system 最前），以免拉开「瞬时块 ↔ 当前 user」的距离。优先在**【三种记忆】块内部**调整顺序与优先级说明。

## 三、外部建议的六点 — 评审结论

| # | 原建议摘要 | 采纳？ | 修正后结论 |
|---|------------|--------|------------|
| 1 | system 越长末尾注意力越差，应整段倒序 | **部分** | 当前瞬时已在 system **末尾**并紧贴 user，近因尚可。风险在**层间冲突**与**总长度截断**。改：块内瞬时优先 + 显式优先级，**不**整段倒序。 |
| 2 | 加「Live2D 数字人」角色锚点 | **是** | 放 system **绝对开头**；可略扩展为一句身份+口吻约束，避免与 persona 矛盾。 |
| 3 | user 尾加「1～3 句回复」 | **是** | 放 **user** 末尾（非 system）；可按人设/env 配置。 |
| 4 | 三种记忆加优先级文案 | **是** | 低成本高收益；与滚动合并、user_profile 分工一致。 |
| 5 | 标签模型等主回复后再跑 | **可选** | 问题真实；并行是为低延迟。建议：**开关**支持「回复后补标签 / 校正」。 |
| 6 | 总上下文优雅降级 | **是** | 在 `_build_memory_for_model` 末尾实现；顺序：砍短期 → 砍长期 → 减瞬时轮数（**保留最新轮**）。需实测 Ollama 截断方向。 |

---

## 四、修改原则

1. **保留**单 system + 单 user + 三层记忆注入架构。
2. **保留** system 宏观顺序：人设/场景/画像/时间 → 记忆块（记忆块仍在 system 尾）。
3. 所有新行为尽量 **env 可关**，便于 A/B 与回滚。
4. 改动集中在 `wschat.py` 拼装层，**不**改 `PersonaRepository`、Redis 写入协议（除非后续单独需求）。
5. 标签模型策略用开关，默认保持现有并行行为，避免无预警增加首包延迟。

## 五、分阶段实施方案

### 阶段 A（建议优先落地，约 10～30 行有效逻辑）

| 编号 | 项 | 说明 |
|------|-----|------|
| A1 | 角色锚点 | system 最前插入 `CHAT_ROLE_ANCHOR`（可 env 覆盖，空则关闭） |
| A2 | 记忆优先级 | 重写 `_format_three_memories_block`：块内顺序 **瞬时 → 短期 → 长期** + 冲突说明一句 |
| A3 | 回复长度 | `ollama_chat_messages` 在 user 尾追加 `CHAT_REPLY_LENGTH_HINT`（一期先用 env 全局） |

### 阶段 B（本周，中等改动）

| 编号 | 项 | 说明 |
|------|-----|------|
| B1 | 总上下文预算 | 新增 `_trim_chat_system_to_budget(sys_content, user_message)`，在 `_build_memory_for_model` 返回前调用 |
| B2 | 预算算法 | 超长时按轮重算：缩短期 → 缩长期（仍保留 long 文尾策略）→ 减少 `instant_turns` 条数（保留最新 N 轮） |
| B3 | 观测日志 | 在现有 `【/ws/chat 记忆装配】` 日志中增加 `system_chars`、`user_chars`、`trimmed=` 标记 |

### 阶段 C（产品确认后，可选）

| 编号 | 项 | 说明 |
|------|-----|------|
| C1 | 标签串行/校正 | `LIVE2D_TAGS_AFTER_REPLY=1`：主流式 `done` 后用「用户输入 + 助手全文」再调标签模型，可选覆盖并行结果 |
| C2 | 人设级长度 hint | 从 `persona.tone_style` 或扩展字段解析「回复句数」，覆盖全局 env |
| C3 | 截断实测文档 | 记录 `OLLAMA_NUM_CTX` / 模型 id 下 Ollama 实际截断行为，修正 B1 策略 |

## 六、具体改动设计（文件级）

### 6.1 `PY/router/wschat.py`

#### （1）新增配置读取

```python
def _chat_role_anchor() -> str:
    # 默认一句；CHAT_ROLE_ANCHOR= 空字符串可关闭
    ...

def _chat_reply_length_hint() -> str:
    # 默认 "\n\n请用 1～3 句话回复。" ；CHAT_REPLY_LENGTH_HINT= 可关闭
    ...

def _chat_prompt_max_chars() -> int:
    # CHAT_PROMPT_MAX_CHARS，默认 7000（system+user 前缀，不含模型生成）
    ...
```

#### （2）`_format_three_memories_block` — 阶段 A2

**块内**新结构示例：

```
【三种记忆】
若以下信息冲突，以优先级从高到低为准：最近对话 > 近期摘要 > 长期脉络。

【最高优先级：最近对话】
用户：…
助手：…
…

【高优先级：近期摘要】
…

【低优先级：长期脉络】
…
```

- 瞬时格式化逻辑不变，仅调整标题与顺序。
- `long_block` / `short_plain` 仍由 `_build_memory_for_model` 传入。

#### （3）`_assemble_main_chat_system_content` — 建议抽取

将当前多处 `"\n\n".join([role_block, scene_block, user_block, time_block, mem_block])` 抽成单一函数，避免 `_build_memory_for_model`、`_guest_main_chat_system_prompt`、`_chat_system_prompt_for_session` 三处漂移：

```python
def _assemble_main_chat_system_content(
    role_block, scene_block, user_block, time_block, mem_block,
) -> str:
    anchor = _chat_role_anchor()
    body = "\n\n".join([role_block, scene_block, user_block, time_block, mem_block])
    return (anchor + "\n\n" + body).strip() if anchor else body
```

#### （4）`_trim_chat_system_to_budget` — 阶段 B1

伪代码：

```
budget = _chat_prompt_max_chars()
user_len = len(user_message) + len(reply_hint)
if len(sys) + user_len <= budget: return sys, meta

# 1) 缩短 short_plain
# 2) 缩短 long_block（保留尾部，与 format_long_memory_block 一致）
# 3) instant_turns = instant_turns[-k:] 递减 k
# 每步后重新 _format_three_memories_block + _assemble...
# 记录 trimmed_steps 写日志
```

**禁止**：对最终 sys 简单 `sys[:budget]` 一刀切（会从头部砍掉锚点或块首）。

#### （5）`ollama_chat_messages` — 阶段 A3

```python
hint = _chat_reply_length_hint()
suffix = hint if hint else ""
msgs.append({"role": "user", "content": f"【当前用户问题】\n{um}{suffix}"})
```

#### （6）Live2D 标签 — 阶段 C1（可选）

- 新增 `_emit_live2d_tags_after_reply(..., assistant_text: str)`
- user content 增加【本轮助手回复（已定稿）】段
- `chat_websocket` 在 `type=done` 且 `LIVE2D_TAGS_AFTER_REPLY=1` 时调用

### 6.2 无需改动的模块（阶段 A/B）

- `memory_layers.py`：Redis 读写、挤出短期逻辑不变。
- `long_memory_fields.py` / `long_memory_consolidator.py`：长期合并逻辑不变。
- `user_profile_refresh.py`：画像格式不变。
- `live2d_catalog.py`：标签 system 可不改；阶段 C 仅改 wschat 侧 user 拼装。

## 七、环境变量（拟新增）

| 变量 | 默认值 | 含义 |
|------|--------|------|
| `CHAT_ROLE_ANCHOR` | 内置一句 Live2D 角色锚点 | system 最前；设为空则关闭 |
| `CHAT_IDENTITY_RULES` | 内置【身份与称呼】规则（角色名≠用户名） | 锚点后；设为空则关闭 |
| `CHAT_REPLY_LENGTH_HINT` | 内置 1～3 句 + 勿自问自答等 | 追加在 user 末尾；设为空则关闭 |
| `CHAT_PROMPT_MAX_CHARS` | `7000` | system + user（含 hint）字符预算上限 |
| `CHAT_PROMPT_TRIM_ENABLED` | `1` | 0=仅告警不裁剪（调试用） |
| `LIVE2D_TAGS_AFTER_REPLY` | `0` | 1=主回复完成后再跑标签（阶段 C） |
| `LIVE2D_TAGS_PARALLEL` | `1` | 与上项配合；默认保持现有并行 |

已有变量继续生效：`LONG_MEMORY_PROMPT_MAX_CHARS`、`SHORT_TERM_PROMPT_MAX_CHARS`、`INSTANT_MEMORY_MAX_TURNS`、`OLLAMA_CHAT_TEMPERATURE` 等。

## 十、风险与回滚

| 风险 | 缓解 |
|------|------|
| 锚点与 persona 语气冲突 | 锚点极短；可 env 关闭 |
| 长度 hint 抑制剧情型人设 | `CHAT_REPLY_LENGTH_HINT` 置空或阶段 C2 按人设配置 |
| 标签延后影响体验 | 默认 `LIVE2D_TAGS_AFTER_REPLY=0` |
| 裁剪过度丢记忆 | 日志标记 trim 步骤；先砍短期再长期 |

**回滚**：将 `CHAT_ROLE_ANCHOR`、`CHAT_REPLY_LENGTH_HINT` 置空，`CHAT_PROMPT_TRIM_ENABLED=0`，即接近现网行为。

## 十二、与长期记忆滚动合并的关系

本方案**只改聊天注入**，不改 `period_overview` 的生成与合并逻辑。长期记忆仍由 `后端项目正文.md` 附录 C §5 / §5.1 描述的后台任务维护；注入侧继续通过 `merge_long_memory_record_for_prompt` + `format_long_memory_block`（文尾保留）进入【低优先级：长期脉络】。

## 十四、身份混淆补丁（2026-05 增补）

**现象**：用户确认「你是西奥」后问「我的名字是？」，模型将角色名「西奥」当作用户名。

**改动**：

| 文件 | 内容 |
|------|------|
| `wschat.py` | 默认 `CHAT_IDENTITY_RULES`：角色人设中的名字≠用户称呼；未写明时应请用户告知 |
| `user_profile_refresh.py` | 画像 LLM：`user_tags` 可写用户自称，禁止写角色名；注入块注明【用户人设】描述真人用户 |
